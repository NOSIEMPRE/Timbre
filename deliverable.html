Part C · 质量自评

三问自评

01

你用什么标准衡量这个 Skill 的输出质量？

判断标准只有一个：这条记录能不能直接进 outreach 流程？我设了三个最低条件：创始人姓名已知，至少一条机构背书信号（YC 批次、融资公告、媒体报道），信号来自两个以上独立来源。三条都满足才进列表，缺一降级或剔除。

在此基础上，eval/quality_check.py 把这套判断量化成了 0–100 的评分，权重最高的两项是节段完整性（40 分，8 个必须节段）和引用密度（25 分，正文 [N] 标注对应真实 URL）。评分低于 75 分时，自动触发 LLM 二次审核，从事实可信度、完整性、精准度、风险质量四个维度打分。平时可以跑 timbre eval，不消耗 API，离线批量扫描全部档案。

02

你做了哪些设计决策来控制"噪声"？

噪声主要来自两类：大公司发布新功能被误当成 startup，以及阶段不在 pre-seed/seed 的项目混入。我做了三层过滤。

  • Query 层：所有 query 都有 "seed"/"2025" 限定词，搜索时用 site: 只看 HN 和 TechCrunch，直接拦截大部分发布会新闻。
  • 提取层：给 LLM 的 prompt 里有四条硬规则，成立时间、阶段、技术差异化、排除已知独角兽，并且强制输出上限 12 个，让模型做取舍，不能全量返回。
  • 展示层：▲●▽ 三档分级，analyst 扫一眼就可以跳过低分项目，只有「高」评级的才建议进 founder-research 深研。
03

如果给你两周时间继续打磨，你优先改进哪一个环节？

目前最大的缺口是 Stage 2.5 的 founder 校验逻辑。现在的做法是：搜索候选姓名、通过正则过滤、写入档案。缺的是最后一步，确认"这个人是不是这家公司的 founder"。Stamp 在两轮测试里都出现了错误归属，搜索结果页上的其他人名被提取进来了。在 Stage 2.5 结束前加一次模型确认，这个问题基本可以解决。

第二件事是跨 session 去重。用几周之后，每次扫描都会重新浮现同一批 YC W25 公司，新鲜感消失，list fatigue 来了。在 SQLite 记忆层加公司指纹，已研究过的往后排，真正新的往前推，这是让工具保持实用价值的关键。两件事都是一两天的工作量，不需要改 pipeline 结构。
