Part C · 质量自评

三问自评

01

你用什么标准衡量这个 Skill 的输出质量？

我在 eval/quality_check.py 中定义了 0–100 分的可量化评分模型，重点衡量四个维度，而非简单统计输出数量：

  • 信号密度：每个项目的 signals 列表是否包含 ≥2 条可溯源信号（YC 批次、TechCrunch 报道、融资公告等），而非泛泛描述。
  • 创始人可识别性：founder 字段是否为真实姓名。这是 VC 实际工作中第一个需要确认的字段，unknown 意味着该项目无法直接进入 outreach 流程。
  • 来源多样性：同一项目是否跨多个数据源出现（HN + TechCrunch + YC = 强信号）。单来源项目在排序中降权。
  • 阶段精准度：stage 字段是否明确为 Pre-Seed / Seed，而非 "Active" 或 "unknown"。阶段模糊说明信息不足，不应进入最终列表。
02

你做了哪些设计决策来控制"噪声"？

噪声有两种：输出量太大（信息过载）和相关性低（非早期项目混入）。我做了三层过滤：

  • Query 层：全部 query 加入 "seed" / "pre-seed" / "2025" 限定词，用 site: 限制来源（HN、TechCrunch），从源头过滤大公司新闻。
  • LLM 提取层：Prompt 给出四条硬筛选规则（成立时间、阶段、技术壁垒、排除独角兽），输出上限强制设为 12 个，让模型做选择而非全量返回。
  • 展示层：按 vc_appeal 用 ▲●▽ 图标显示优先级，用户可快速跳过低分项目，无需读完全列表。只有「高」评级项目建议进入 founder-research 深研。
03

如果给你两周时间继续打磨，你优先改进哪一个环节？

优先改进 Stage 2 的创始人识别准确率。当前输出中 founder 字段约 22% 仍为 unknown（已比初版的 60% 大幅改善），这是对 VC 实用性影响最大的缺口——有具名创始人的项目是最容易付诸行动的。

本次交付中已实现了 Stage 2.5（定向创始人查找），通过对 unknown 项目补发精准 web_search + 多层正则清洗（过滤月份缩写、公司后缀、URL 片段等），将识别率从 ~40% 提升到 ~78%。

下一步改进方向：在 Stage 2.5 之后增加一次 LLM 交叉验证——将找到的候选姓名和公司名一起发回给模型，确认"这个人是否真的是这家公司的创始人"，消除剩余的错误归属（如 Miyagi Labs 的创始人被误分配到 Stamp 的情况）。
